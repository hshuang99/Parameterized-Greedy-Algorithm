\chapter{Our Greedy Algorithms}
\label{ch:our-algorithms}
In this section, we introduce our methods for synthesizing low-depth CNOT circuits for block ciphers. Our approach is based on Shi and Feng's greedy method, specifically for block sizes of $16 \times 16$ and $32 \times 32$, as discussed in \cite{cryptoeprint:2023/165}.

We propose a parameterized, depth-oriented algorithm for synthesizing ancilla-free, low-depth CNOT circuits, and we develop two distinct greedy strategies to further explore depth-optimized results.

By utilizing different matrix representations and heuristic search strategies, we implement the Row-or-Column and Parallel greedy algorithms. The implementation logic and pseudocode for each approach are detailed in the following sections.

\section{Elements}
In this section, we introduce some elements that will be used in this work.

\subsection{Variables}

We show all of variables used in our methods.

\begin{itemize}
    \item $M$: Given an $n \times n$ input matrix $A$, our greedy algorithm seeks to identify a sequence of operations that transforms $A$ into a permutation matrix.
    \item $M'$: The matrix cost function is derived from the inverse of matrix $A$.
    \item $d$: The circuit depth achieved when matrix $A$ is transformed into a permutation matrix.
    \item SIZE: Since we are specifically considering square block matrices, the matrix size is determined by retrieving the row or column length using the built-in `length()` function.
    \item MAX: Since our approach will explore all possible operations until $M$ become to permutation, we would like start from maximum reduce to minimum so that we choose system.float\_info.max as our condition.
    \item $L_{r}, L_{c}$: Row and column operation sequences are stored in separate layer lists for each iteration.
    \item $Ls_{r}, Ls_{c}$: We maintain row-layer and column-layer lists to store the specific operations assigned to each discrete circuit layer.
    \item $L_{row}, L_{col}$: A list of remaining operations is updated after each step to exclude those already applied to the matrix.
    \item $L_{row}^{cost}, L_{col}^{cost}$: For every operation in $L_{row}$ and $L_{col}$, the algorithm calculates and stores the resulting matrix cost function in a lookup list.
    \item $OPs_{row}, OPS_{col}$: The collection of applied row and column operations is stored to construct the final sequence list and to verify the correctness of the transformation.
    \item LATEST\_MINM\_DEPTH: As the name, it is using for record the latest minimum depth, if the depth over this size, then we setting "over" to True.
    \item $p$: A control parameter is used to select the matrix cost function type. Supported functions include linear sum (1), square (2), origin (3) and logarithmic (- 1), which can be selected via their respective numerical identifiers.
    \item over: An acceptance flag is utilized to evaluate the current circuit depth; it returns 'True' if the depth $d$ exceeds the current minimum depth, and 'False' otherwise.
    \item close\_permu: A termination flag monitors whether the matrix is nearing a permutation state. If the current matrix can be transformed into a permutation matrix within a single layer, the flag is set to True; otherwise, it remains False.
    \item minm: The algorithm identifies the current minimum cost across all candidates and selects the operation corresponding to this optimal value.
    \item origin, inverse: At the start of the process, a copy of the input matrix is retained for final verification. Simultaneously, the matrix inverse is calculated to facilitate the evaluation of the cost function.
    \item $visi_{row}, visi_{col}$: A visited list tracks the selected row and column indices; once a position is chosen, it is excluded from future iterations to prevent redundant operations.
    \item select\_list: The selection list stores the set of candidate operations available for the current optimization step.
    \item $H_{r}, H_{c}$: The matrix cost function is evaluated from both a row perspective and a column perspective to determine the most efficient operation.
    \item $B_{row}, B_{col}$: The algorithm maintains a collection of row and column operations for both the Row/Column\-wise and Parallel greedy approaches; it identifies the available operators and selects the one with the lowest computational cost.
    \item $best_{row}^{cost}, best_{col}^{cost}$: The algorithm maintains a collection of cost values for each potential operation in $B_{row}$ and $B_{col}$ to facilitate the selection of the optimal operator.
    \item stuck\_counter: A counter is used to record the number of consecutive stalls (stuck iterations) to detect when the algorithm has reached a local optimum.
    \item max\_stuck\_iteration: A stagnation threshold is defined to limit the number of consecutive stalls; by default, this value is set to 3.
    \item Candidates: The algorithm also maintains a collection of non-optimal operators, representing candidates whose cost values exceeded the current minimum.
    \item best\_escape: To resolve local minima, the algorithm identifies the optimal candidate from the escapeCandidates list—after sorting it in ascending order—and adds it to the select\_list to resume the search.
    \item minm\_size: A threshold condition is implemented to ensure that the circuit depth (total number of layers) does not exceed the predefined limit.
    \item reduce: A copy of the initial matrix is retained to verify that the sequence of row and column operations ($row_{op}$ and $col_{op}$) successfully transforms the origin matrix into the target representation.
    \item size\_minm: The algorithm records the final circuit depth to document the efficiency of the optimized sequence.
    \item per: The algorithm records the final matrix state to generate a permutation list, which defines the final mapping of the quantum registers.
    \item seq: The algorithm maintains a comprehensive log of all applied row and column operations ($row_{op}$ and $col_{op}$), which constitutes the final quantum circuit sequence.
    \item Layers: The algorithm maintains a record of the sets $Ls_{c}$ and $Ls_{r}$, which contain the column and row operations assigned to each parallel layer.
    \item correct: A validation flag is employed to confirm that the combined operation sequence and layer assignments successfully transform the original matrix into a permutation matrix.
\end{itemize}

\subsection{Cost Functions}

In this section, we introduce the cost functions used in this work. Later, in Algorithm~\ref{alg:costMat}, the global cost of the current matrices $M$ and $M'$ is determined as a function of $p$; locally, we adopt Algorithm~\ref{alg:functionCost} to compute the cost with respect to direction, likewise depending on the value of $p$.

\begin{algorithm}[htbp]
\caption{costMat}
\label{alg:costMat}
\begin{algorithmic}[1]
\Require $M/M', p$
\Ensure Calculate the cost value for each cost function depends on $p$ value
\State cost $\leftarrow \varnothing$
\State $n \leftarrow $ Length($M/M'$)
\Match{$p$}
	\Case{prod}
		\State cost $\leftarrow \sum_{i=0}^{n} log_{2}(\sum_{j=0}^{n} M_{(i, j)}/M'_{(i, j)})$
	\EndCase
	\Case{sum}
		\State cost $\leftarrow \sum_{i=0}^{n}(\sum_{j=0}^{n}M_{(i, j)}/M'_{(i,j)}$)
	\EndCase
	\Case{square}
		\State cost $\leftarrow \sum_{i=0}^{n}(\sum_{j=0}^{n}M_{(i,j)}/M'_{(i,j)})^{2}$
	\EndCase
	\Case{origin}
		\State cost $\leftarrow \sum_{i=0}^{n}\sum_{j=0}^{n}(M_{(i,j)}/M'_{(i,j)} \oplus I_{n \times n})$
	\EndCase
\EndMatch
\State \Return cost
\end{algorithmic}
\end{algorithm}


\begin{algorithm}[htbp]
\caption{functionCost}
\label{alg:functionCost}
\begin{algorithmic}[1]
\Require $M, M', p$
\Ensure Calculate the minimum cost for current $M$ and $M'$ depends on $p$ value
\If{$p \neq 1 \hspace{2mm} || \hspace{2mm} 3$}
	\State $H_{r}$ = costMat($M, p$) + costMat($M', p$)
	\State $H_{c}$ = costMat($M^{T}, p$) + costMat($M', p$)
	\State cost $\leftarrow$ max($H_{r}, H_{c}$)
\Else
	\State cost $\leftarrow$ costMat($M, p$) + costMat($M', p$)
\EndIf
\State \Return cost
\end{algorithmic}
\end{algorithm}

\subsection{Pre-Processing}

To collect the set of operations available for circuit synthesis, we follow the principle that an operator cannot be selected if its row or column has already been visited, i.e.\ if it appears in $visi_{row}$ or $visi_{col}$. This constraint ensures efficient use of the quantum circuit area: for each layer, we aim to place as many operators as possible. Our tables $L_r$ and $L_c$ are constructed to guarantee this property, as described in Algorithm~\ref{alg:L-collection}.

\begin{algorithm}[htbp]
\caption{L\_Collection}
\label{alg:L-collection}
\begin{algorithmic}[1]
\Require $visi_{row/col}$, SIZE
\Ensure Capture all of available operators in large table $L$ depends on row or column
	\For{$i = 0$ \textbf{to} SIZE}
		\If{$visi_{row/col}[i] == 1$}
			\State \textbf{continue}
		\For{$j = 0$ \textbf{to} SIZE}
			\If{$visi_{row/col}[i] == 1$ \textbf{or} $j == i$}
				\State \textbf{continue}
			\EndIf
			\State $L$.append($i, j$)
		\EndFor
		\EndIf
	\EndFor
\State \Return $L$
\end{algorithmic}
\end{algorithm}

\subsection{Operator Selection}

After collecting all valid operators into our table $L$, we select a subset of operations to constitute a single-layer circuit. The core idea is that every selected operation should reduce the value of the cost function when applied, as shown in Algorithm~\ref{alg:ops-sel}. We use the function row\_i2j to execute the type-1 elementary row (or column) operation of adding one row (or column) to another, and then place the result into the target matrix. Note that row\_i2j is applied to $M$ and col\_i2j to $M'$ when the operator originates from a row; the roles are exchanged when the operator originates from a column.

The cost value of every available operator is stored in $L_{row}^{cost}$ or $L_{col}^{cost}$, depending on the operator type. An operator is selected, added to "select\_list" to form part of the current layer, and the minimum cost is updated accordingly, if its cost value is lower than the current minimum. We also accept operators whose cost leaves the cost function unchanged (i.e.\ equal to the current value), but in this case the minimum cost is not updated.

Occasionally, the search falls into a local minimum. To address this, we introduce a stuck threshold: once this threshold is reached, we switch to Algorithm~\ref{alg:rand-ops-sel}, which no longer requires the cost to decrease. Instead, we allow the cost to increase temporarily, which may help the search escape the local minimum.

\begin{algorithm}[htbp]
\caption{Ops\_Sel}
\label{alg:ops-sel}
\begin{algorithmic}[1]
\Require $L_{row/col}, L_{row/col}^{cost}, M, M', p$, minm
\Ensure Select all of available operators for a single layer depends on Row or Column
\State $n \leftarrow$ Length($L_{row/col}$)
\For{$i = 0$ \textbf{to} $n$}
	\State tmp\_mat $\leftarrow$ row\_i2j($M, L_{row/col}[i].row, L_{row/col}[i].col$) \Comment{or col\_i2j}
	\State tmp\_inv $\leftarrow$ col\_i2j($M', L_{row/col}[i].col, L_{row/col}[i].row$) \Comment{or row\_i2j}
	\State cost $\leftarrow$ OperationCost(tmp\_mat, tmp\_inv, $p$)
	\State $L_{row/col}^{cost}$.append(cost)
\EndFor
\For{$i = 0$ \textbf{to} $n$}
	\If{$L_{row/col}^{cost}[i].cost < $ minm}
		\State select\_list.clear()
		\State select\_list.append($L_{row/col}^{cost}[i]$.row, $L_{row/col}^{cost}[i]$.col)
		\State minm $\leftarrow L_{row/col}^{cost}$.cost
	\ElsIf{$L_{row/col}^{cost}[i] == $ minm}
		\State select\_list.append($L_{row/col}^{cost}[i]$.row, $L_{row/col}^{cost}[i]$.col)
	\EndIf
\EndFor
\State \Return select\_list, minm
\end{algorithmic}
\end{algorithm}

\begin{algorithm}[htbp]
\caption{Rand\_Ops\_Sel}
\label{alg:rand-ops-sel}
\begin{algorithmic}[1]
\Require $L_{row/col}, M, M', p$
\Ensure Select some of random operators for a single layer depends on Row or Column and not require less than minimum cost
\State $n \leftarrow$ Length($L_{row/col}$)
\For{$i = 0$ \textbf{to} $n$}
	\State tmp\_mat $\leftarrow$ row\_i2j($M, L_{row/col}[i].row, L_{row/col}[i].col$) \Comment{or col\_i2j}
	\State tmp\_inv $\leftarrow$ col\_i2j($M', L_{row/col}[i].col, L_{row/col}[i].row$) \Comment{or row\_i2j}
	\State cost $\leftarrow$ OperationCost(tmp\_mat, tmp\_inv, $p$)
	\State Candidates.append(cost)
\EndFor
\State \Return Candidates
\end{algorithmic}
\end{algorithm}

\subsection{Hamming-Weight Check}

This is one of our strategies for finding improved results in Algorithm~\ref{alg:can-close-permu}. We check whether $M$ has become a permutation matrix; since our goal is to find the shortest path to a permutation (identity) matrix, we check whether the Hamming weight of each row and column of $M$ is less than or equal to $2$.

\begin{algorithm}[htbp]
\caption{Can\_Close\_Permu}
\label{alg:can-close-permu}
\begin{algorithmic}[1]
\Require $M$, close\_permu
\Ensure Check the $M$ whether close to permutation matrix
\State $n \leftarrow$ Length($M$)
\For{$i = 0$ \textbf{to} $n$}
	\If{$\sum M[i, :] > 2$}
		\State \Return $False$
	\EndIf
\EndFor
\For{$i = 0$ \textbf{to} $n$}
	\If{$\sum M[:, i] > 2$}
		\State \Return $False$
	\EndIf
\EndFor
\State \Return $True$
\end{algorithmic}
\end{algorithm}

\subsection{Operation Execution}

For the operations obtained from select\_list, we execute them on $M$ and $M'$ in Algorithm~\ref{alg:ops-exec}, randomly choosing an operator that reduces the cost value; this choice is also recorded in $visi_{row}$ or $visi_{col}$ to guarantee that the corresponding circuit position is not used again.

Another condition arises when select\_list is empty, meaning that all positions in the current layer have been filled. In this case, we transfer $L_r$ into $Ls_r$ and $L_c$ into $Ls_c$ to finalize the synthesis for this layer, and then reset the visited lists before proceeding to the next round.

\begin{algorithm}[htbp]
\caption{Ops\_Exec}
\label{alg:ops-exec}
\begin{algorithmic}[1]
\Require select\_list, $M, M', visi_{row/col}$, close\_permu
\Ensure Execution available operations from select\_list depends on $visi_{row/col}$ address
\If{|select\_list| $ == 0$}
	\If{$|L_{row/col}| > 0$}
		\State $Ls_{row/col}$.append($L_{row/col}$)
		\State $L_{row/col}$.clear()
		\State $visi_{row/col}$.clear()
	\EndIf
	\If{can\_close\_permu($M$)}
		\State close\_permu $\leftarrow True$
	\EndIf
\Else
	\State rand $\leftarrow$ random()
	\State sel\_op $\leftarrow$ select\_list[rand]
	\State $M \leftarrow$ row\_i2j($M$, sel\_op.row, sel\_op.col)
	\State $M' \leftarrow$ col\_i2j($M'$, sel\_op.col, sel\_op.row)
	\State $L_{row/col}$.append(sel\_op.row, sel\_op.col)
	\If{$\sum visi_{row/col} == 0$}
		\State $d$++
	\EndIf
	\State $visi_{row/col}$[sel\_op.row] $\leftarrow 1$
	\State $visi_{row/col}$[sel\_op.col] $\leftarrow 1$
\EndIf
\State \Return select\_list, minm, $M, M', layer_{row/col}, Layers_{row/col}, visi_{row/col}, d$, close\_permu
\end{algorithmic}
\end{algorithm}

\subsection{Verifier}

As Algorithm~\ref{alg:ops-exec} is executed round after round, we check whether the remaining operations are still necessary, as described in Algorithm~\ref{alg:last-check}, and extract them into $Ls_r$ or $Ls_c$ accordingly. After clearing the remaining operations, we also verify, in Algorithm~\ref{alg:verify-operations}, that all operations indeed transform $M$ into a different matrix.

We then globally verify all operators using Algorithm~\ref{alg:verify-all}, beginning with Algorithm~\ref{alg:conflict-check}, which ensures that every operation within a layer is distinct. Once we have confirmed that no operations are duplicated, we execute them on a copy of $M$ to verify that they indeed reproduce the final state of $M$.

\begin{algorithm}[htbp]
\caption{Last\_Check}
\label{alg:last-check}
\begin{algorithmic}[1]
\Require $L_{r}, L_{c}$
\Ensure Find the last round operations and insert them to $Ls_{r}$ and $Ls_{c}$
\If{$|L_{r}| > 0$}
	\State $Ls_{r}$.append($L_{r}$)
	\State $L_{r}$.clear()
\EndIf
\If{$|L_{c}| > 0$}
	\State $Ls_{c}$.append($L_{c}$)
	\State $L_{c}$.clear()
\EndIf
\State \Return $Ls_{r}, Ls_{c}$
\end{algorithmic}
\end{algorithm}

\begin{algorithm}[htbp]
\caption{Verify\_Matrix}
\label{alg:verify-operations}
\begin{algorithmic}[1]
\Require $OPs_{row}, OPs_{col}, M$
\Ensure Check operations from OPs that can work on $M$
\State reduce $\leftarrow$ origin
\State size $\leftarrow 0$
\For{$i = 0$ \textbf{to} $|OPs_{row}|$}
	\State reduce $\leftarrow$ row\_i2j(reduce, $OPs_{row}[i]$.row, $OPs_{row}[i]$.col)
	\State size++
\EndFor 
\For{$j = 0$ \textbf{to} $|OPs_{col}|$}
	\State reduce $\leftarrow$ col\_i2j(reduce, $OPs_{col}[j]$.row, $OPs_{col}[j]$.col)
	\State size++
\EndFor
\State ok $\leftarrow True$
\If{reduce $\neq M$}
	\State ok $\leftarrow False$
\EndIf 
\State \Return ok
\end{algorithmic}
\end{algorithm}


\begin{algorithm}[htbp]
\caption{Conflict\_Check}
\label{alg:conflict-check}
\begin{algorithmic}[1]
\Require $Layers$
\Ensure Check operators in Layers are distinct
\State conflict $\leftarrow False$
\For{$i = 0$ \textbf{to} $|Layers|$}
	\State used $\leftarrow [\varnothing]$
	\For{$j = 0$ \textbf{to} $|Layers[i]|$}
		\If{$Layers[i][j]$.row || $Layers[i][j]$.col $\in$ used}
			\State \Return $False$
		\EndIf
		\State used.append($Layers[i][j]$.row)
		\State used.append($Layers[i][j]$.col)
	\EndFor
\EndFor
\State \Return $True$
\end{algorithmic}
\end{algorithm}

\begin{algorithm}[htbp]
\caption{Verify\_All}
\label{alg:verify-all}
\begin{algorithmic}[1]
\Require origin, Layers, seq, $M$
\Ensure Check all operations are work
\State $n \leftarrow |M|$
\If{!Conflict\_Check(Layers)}
	\State \Return $False$
\EndIf
\For{$i = 0$ \textbf{to} |Layers|}
	\State layer\_visi $\leftarrow [\varnothing]$
	\For{$j = 0$ \textbf{to} |Layers[i]|}
		\If{layer\_visi[Layers[i][j].row] || layer\_visi[Layers[i][j].col]}
			\State \Return $False$
		\Else
			\State layer\_visi[Layers[i][j].row] $\leftarrow 1$
			\State layer\_visi[Layers[i][j].col] $\leftarrow 1$
		\EndIf
	\EndFor
\EndFor
\For{$k = 0$ \textbf{to} |seq|}
	\If{seq[k].op == row}
		\State origin $\leftarrow$ col\_i2j(origin, seq[k].col, seq[k].row)
	\Else
		\State origin $\leftarrow$ col\_i2j(origin, seq[k].row, seq[k].col)
	\EndIf
\EndFor
\If{origin == $M$}
	\State \Return $True$
\Else
	\State \Return $False$
\EndIf
\end{algorithmic}
\end{algorithm}

\subsection{Record}

Finally, we record all relevant data, including the permutation list, the operation sequence, and the synthesized layers in Algorithm~\ref{alg:generator}. Row operators are reversed in the same manner as column operators, and the permutation list is used to record their addresses.

\begin{algorithm}[htbp]
\caption{Generator}
\label{alg:generator}
\begin{algorithmic}[1]
\Require $M, OPs_{row}, OPs_{col}, Ls_{r}, Ls_{c}$
\Ensure Collection permutation list, sequence list and Layers 
\State per, seq, Layers $\leftarrow [\varnothing]$
\For{$i = 0$ \textbf{to} $|M|$}
	\For{$j = 0$ \textbf{to} $|M|$}
		\If{$M[i][j]$}
			\State per[$i$] = $j$
		\EndIf
	\EndFor
\EndFor 
\For{$i = 0$ \textbf{to} $|OPs_{col}|$}
	\State seq.append($OPs_{col}[i]$.row, $OPs_{col}[i]$.col)
\EndFor
\State $OPs_{row} \leftarrow$ reverse($OPs_{row}$)
\For{$j = 0$\ \textbf{to} $|OPs_{row}|$}
	\State seq.append(per[$OPs_{row}[j]$.row], per[$OPs_{row}[j]$.col])
\EndFor
\For{$i = 0$ \textbf{to} $|Ls_{c}|$}
	\State nl\_c $\leftarrow [\varnothing]$
	\For{$j = 0$ \textbf{to} $|Ls_{c}[i]|$}
		\State nl\_c.append($Ls_{c}[i][j]$.row, $Ls_{c}[i][j]$.col)
	\EndFor
	\State Layers.append(nl\_c)
\EndFor
\State $Ls_{r} \leftarrow$ reverse($Ls_{r}$)
\For{$k = 0$ \textbf{to} $|Ls_{r}|$}
	\State nl\_r $\leftarrow [\varnothing]$
	\For{$l = 0$ \textbf{to} $|Ls_{r}[k]|$}
		\State nl\_r.append(per[$Ls_{r}[k][l]$.row], per[$Ls_{r}[k][l]$.col])
	\EndFor
	\State Layers.append(nl\_r)
\EndFor
\State \Return per, seq, Layers
\end{algorithmic}
\end{algorithm}

\section{Improved Shi and Feng's Work}

In this section, we clarify Shi and Feng's work by presenting it as pseudocode in Algorithm~\ref{alg:Modified-Shi-Feng-Greedy}. We modify their strategy so that operators which do not reduce the cost are not selected initially; only once the stuck threshold is exceeded do we accept operations that increase the cost function. Additionally, in their "can\_one" condition, they consider only column operators for collection; based on our observations, we instead change the cost function in this condition to Table~\ref{tab:product-hamming-weight}, which guides $M$ when it is nearly complete, toward the correct direction more quickly. The effect of this modification can be seen in our experimental results (Chapter~\ref{ch:experimental-results}).

\begin{algorithm}[htbp]
\caption{Row/Col Greedy}
\label{alg:Modified-Shi-Feng-Greedy}
\begin{algorithmic}[1]
\Require $M \in \mathbb{F}_{2}^{n \times n}$, $M' \in \mathbb{F}_{2}^{n \times n}$, $L_{r}, L_{c}, Ls_{r}, Ls_{c}, p$ and over
\Ensure $Ls_{r}, Ls_{c}, row_{op}, col_{op}$ that implement permutation matrix with length $d$
\State SIZE $\leftarrow$ $|M|$
\State minm $\leftarrow$ Max
\State select\_list $\leftarrow$ list($\varnothing$)
\State $visi_{row}$ $\leftarrow [\varnothing]$, $visi_{col}$ $\leftarrow [\varnothing]$
\State close\_permu $\leftarrow$ False
\State LIMIT $\leftarrow$ LATEST\_MINM\_DEPTH
\While{$M$ != Permutation}
	\State select\_list.clear()
	\State $L_{row/col}$.clear(), $L_{row/col}^{cost}$.clear()
	
	\State $H_{r}, H_{c}$, minm $\leftarrow$ functionCost($M, M', p$)
    
    \If{!close\_permu}
		\State $L_{col} \leftarrow L\_Collection(visi_{row})$
	    \State select\_list, minm $\leftarrow Ops\_Sel( L_{col}, L_{col}^{cost}, M, M', p)$ 
    \EndIf

	\State $L_{row} \leftarrow L\_Collection(visi_{row}$)
	\State select\_list, minm $\leftarrow$ Ops\_Sel($L_{row}, L_{row}^{cost}, M, M', p$)
    
	\State $L_{r/c}, Ls_{r/c}, M, M', OPs_{row/col}, visi_{row/col}, d$, over $\leftarrow$ Ops\_Exec()

	\If{close\_permu}
		\State $p \leftarrow -1$
	\EndIf
    
    \If{$d >$  LIMIT}
	    \State over $\leftarrow True$ 
	    \State \Return
    \EndIf
\EndWhile
\State \Return $L_{r/c}, Ls_{r/c}, M, OPs_{row/col}, d$, over
\If{close\_permu}
	\State $p \leftarrow -1$
\EndIf
\end{algorithmic}
\end{algorithm}

\section{Hamming-Weight Guide Strategy}

Our cost function is derived from the greedy framework proposed by \cite{schaeffer2014cost}, which uses the Hamming distance to the identity matrix, computed row-wise, as follows:

\begin{equation}
	h_{origin}(M) = \sum_{i=1}^{n}\sum_{j=1}^{n}(M_{(i,j)} \oplus I_{(i,j)}) + (M^{-1}_{(i,j)} \oplus I_{(i,j)})
\end{equation}

and we use this function to guide our search path. As mentioned in Section~\ref{sec:CNOT-transformation-to-Matrix-Operations}, CNOT synthesis can be regarded in terms of type-1 and type-3 elementary matrices, and according to Theorem~\ref{eq:elementary-matrix-operation-type-1-and-3}, we can expand this into the following equations:

\begin{equation}
	\label{eq:elementary-matrix-left-multiplication}
	R_{k} \cdots R_{1} \cdot M = P \Rightarrow M = R_{1}^{-1} \cdots R_{k}^{-1} \cdot P
\end{equation}

\begin{equation}
	\label{eq:elementary-matrix-left-right-multiplication}
	R_{k} \cdots R_{1} \cdot M \cdot C_{1} \cdots C_{k'} = P \Rightarrow M = R_{1}^{-1} \cdots R_{k}^{-1} \cdot P \cdot C_{k'}^{-1} \cdots C_{1}^{-1}
\end{equation}

where $P$ is a permutation matrix, and we can synthesize the operators $R$ and $C$ using our $h_{origin}$ cost function. In some cases, $R$ and $C$ can be interchanged, which means our approach need not be concerned with direction when computing the cost function; this effect is even more beneficial for certain block ciphers or MDS matrices.

\section{Row-or-Col}
The first strategy we evaluates both row and column operations and selects the layer that yields the minimum matrix cost. In cases where the costs are identical, the algorithm defaults to the column-wise operation. If only one set of operations is available, the algorithm selects the existing set. 

To address the inherent risk of local minima in greedy heuristics, we implement a stuck threshold. If the cost fails to decrease for three consecutive iterations, the algorithm accepts a sub-optimal operation that increases the cost, thereby enabling an escape from the local minimum. Detailed in Algorithm~\ref{alg:row-or-col-greedy}.

\begin{algorithm}[htbp]
\caption{Row-or-Col Greedy}
\label{alg:row-or-col-greedy}
\footnotesize
\begin{algorithmic}[1]
\Require $M \in \mathbb{F}_{2}^{n \times n}$, $M' \in \mathbb{F}_{2}^{n \times n}$, $L_{r}, L_{c}, Ls_{r}, Ls_{c}, p$, over
\Ensure $Ls_{r}, Ls_{c}, row_{op}, col_{op}$ that implement permutation matrix with length $d$
\State SIZE $\leftarrow$ $|M|$
\State minm, $best_{row}^{cost}, best_{col}^{cost}$ $\leftarrow$ MAX
\State select\_list $\leftarrow$ list($\varnothing$)
\State $visi_{row}$ $\leftarrow [\varnothing]$, $visi_{col}$ $\leftarrow [\varnothing]$
\State LIMIT $\leftarrow$ LATEST\_MINM\_DEPTH
\State stuck\_counter $\leftarrow$ 0, max\_stuck\_iterations $\leftarrow$ 3
\While{$M$ != Permutation}
	\State select\_list.clear()
	\State $L_{row/col}$.clear(), $L_{row/col}^{cost}$.clear(), $B_{row/col}$.clear()
	
	\State $H_{r}, H_{c}$, minm $\leftarrow$ functionCost($M, M', p$)
    
	\State $L_{row} \leftarrow$ L\_Collection($visi_{row}$)
	\State $B_{row}, best_{row}^{cost} \leftarrow$ Ops\_Sel($L_{row}, L_{row}^{cost}, M, M', p$, minm)
	
	\State $L_{col} \leftarrow$ L\_Collection($visi_{col}$)
	\State $B_{col}, best_{col}^{cost} \leftarrow$ Ops\_Sel($L_{col}, L_{col}^{cost}, M, M', p$, minm)
	
	\State is\_stuck = $|B_{row}|$ == 0 $\land$ $|B_{col}|$ == 0
	
	\If{is\_stuck}
		\State stuck\_counter += 1
	\Else
		\State stuck\_counter $\leftarrow$ 0
    \EndIf
    
    \If{(stuck\_counter $\geq$ max\_stuck) || (is\_stuck $\land \sum{visi_{row}}$ == 0 $\land \sum{visi_{col}}$ == 0)}
		\State Candidates.clear()
		\State Candidates $\leftarrow$ Rand\_Ops\_Sel($L_{row/col}, M, M', p$)
    \EndIf
    
	\If{$best_{row}^{cost} < best_{col}^{cost}$}
		\State select\_list $\leftarrow$ $B_{row}$
		\State minm $\leftarrow$ $best_{row}^{cost}$
	\ElsIf{$best_{row}^{cost} > best_{col}^{cost}$}
		\State select\_list $\leftarrow$ $B_{col}$
		\State minm $\leftarrow$ $best_{col}^{cost}$
	\Else
		\State select\_list $\leftarrow$ $B_{col}$ if $|B_{col}| > 0 $ else $B_{row}$
		\State minm $\leftarrow$ $best_{col}^{cost}$ if |select\_list| $> 0$ else minm
    \EndIf
    
	\State $L_{r/c}, Ls_{r/c}, M, M', OPs_{row/col}, visi_{row/col}, d$, over $\leftarrow$ Ops\_Exec()
    
    \If{$d$ > LIMIT}
	    \State over $\leftarrow True$ 
	    \State \Return
    \EndIf
\EndWhile
\State \Return $L_{r/c}, Ls_{r/c}, M, OPs_{row/col}, d$, over
\end{algorithmic}
\end{algorithm}

\section{Parallel}
Then the second is Parallel Greedy Algorithm~\ref{alg:parallel-greedy} strategy aggregates both row and column operations into a unified `select\_list`. 

In cases where one operation type is unavailable, the algorithm proceeds with the existing candidates. Similar to the Row-or-Column approach, a stagnation threshold is utilized to escape local minima; if no progress is made for three iterations, the algorithm accepts cost-increasing moves. 

The primary distinction of the Parallel strategy lies in its late-stage optimization: when the matrix approaches a permutation state, the 'close\_permu' flag is activated, restricting the search to $h_{prod}(M)$ cost function to finalize the circuit.

\begin{algorithm}[htbp]
\caption{Parallel Greedy}
\label{alg:parallel-greedy}
\footnotesize
\begin{algorithmic}[1]
\Require $M \in \mathbb{F}_{2}^{n \times n}$, $M' \in \mathbb{F}_{2}^{n \times n}$, $L_{r/c}, Ls_{r/c}, p$, over
\Ensure $Ls_{r/c}, OPs_{row/col}$ that implement permutation matrix with length $d$
\State SIZE $\leftarrow |M|$
\State minm, $best_{row}^{cost}, best_{col}^{cost}$ $\leftarrow$ MAX
\State select\_list $\leftarrow$ list($\varnothing$)
\State $visi_{row}$ $\leftarrow$ [$\varnothing$], $visi_{col}$ $\leftarrow$ [$\varnothing$]
\State LIMIT $\leftarrow$ LATEST\_MINM\_DEPTH
\State over $\leftarrow False$ 
\State stuck\_counter $\leftarrow$ 0, max\_stuck\_iterations $\leftarrow$ 3
\While{$M$ != Permutation}
	\State select\_list.clear()
	\State $L_{row/col}$.clear(), $L_{row/col}^{cost}$.clear(), $B_{row/col}$.clear()
	
	\State $H_{r}, H_{c}$, minm $\leftarrow$ functionCost($M, M', p$)
    
    \If{!close\_permu}
		\State $L_{row} \leftarrow$ L\_Collection($visi_{row}$)
		\State $B_{row}, best_{row}^{cost} \leftarrow$ Ops\_Sel($L_{row}$, $L_{row}^{cost}$, $mat$, $inv$, $p$, minm) 
    \EndIf
    
	\State $L_{col} \leftarrow$ L\_Collection($visi_{col}$)
	\State $B_{col}, best_{col}^{cost} \leftarrow$ Ops\_Sel($L_{col}$, $L_{col}^{cost}$, $mat$, $inv$, $p$, minm)
	
	\State is\_stuck = $|B_{row}|$ == 0 $\land$ $|B_{col}|$ == 0
	
	\If{is\_stuck}
		\State stuck\_counter += 1
	\Else
		\State stuck\_counter $\leftarrow$ 0
    \EndIf
    
    \If{(stuck\_counter $\geq$ max\_stuck\_iterations) || (is\_stuck $\land \sum{visi_{row}}$ == 0 $\land \sum{visi_{col}}$ == 0)}
	    \State Candidates.clear()
		\State Candidates $\leftarrow$ Rand\_Ops\_Sel($L_{row/col}, M, M', p$)
	\Else
		\State select\_list $\leftarrow$ $B_{row} + B_{col}$
		\If{$best_{row}^{cost} < best_{col}^{cost}$}
			\State minm $\leftarrow$ $best_{row}^{cost}$
		\ElsIf{$best_{row}^{cost} > best_{col}^{cost}$}
			\State minm $\leftarrow$ $best_{col}^{cost}$
		\Else
			\State minm $\leftarrow$ $best_{col}^{cost}$ if |select\_list| $> 0$ else minm
    	\EndIf
	\EndIf
    
	\State $L_{r/c}, Ls_{r/c}, M, M', OPs_{row/col}, visi_{row/col}, d$, over $\leftarrow$ Ops\_Exec()
	
	\If{close\_permu}
		\State $p \leftarrow -1$
	\EndIf
    
    \If{$d$ > LIMIT}
	    \State over $\leftarrow True$
	    \State \Return
    \EndIf
\EndWhile
\State \Return $L_{r}, L_{c}, Ls_{r}, Ls_{c}, mat, row_{op}, col_{op}, d$, flag
\end{algorithmic}
\end{algorithm}
